# MVP — Codex Web UI (ChatGPT-like) on a VPS

## One-liner
A self-hosted, ChatGPT-style web interface that controls **Codex running on your VPS** (via app-server), with simple email/password login, lockouts against brute force, and full conversation history (new + resume).

Public-repo target: a simplified Open CLA style experience that users can deploy on a VPS and use comfortably from desktop or phone.

User experience target: clone repo, configure `.env`, run `docker compose up -d`, open one HTTPS URL, and start chatting from desktop or mobile.

---

## Risk Warning
Use at your own risk.

This system can perform powerful operations on VPS files/workspace through Codex. Unsafe prompts, misconfiguration, or high-trust execution modes can lead to destructive actions, including accidental deletion of important data.

Minimum safety expectations:
- maintain regular backups/snapshots
- use least-privilege runtime/users
- keep Codex app-server private (never public)
- reserve high-risk modes (for example YOLO/no-sandbox) for trusted environments only
- In multi-user/shared-VPS mode, assume chats/files/workspace are not private between users

## Goals

### Must-have (MVP)
- Chat UI that looks/feels like ChatGPT:
  - Left sidebar: conversation list + “New chat”
  - Main pane: messages, streaming responses, code blocks, copy buttons
  - Input box with multiline + send
  - Fully responsive on desktop + mobile (ChatGPT-like behavior on both)
- Authentication:
  - Email + password login
  - Progressive lockouts after repeated failures (fail2ban-ish)
- Conversations:
  - Start new conversation
  - See past conversations
  - Resume an existing conversation (continue context)
  - Shared visibility model: all users on the VPS can view shared conversation history/files
  - Concurrency safety: only one active Codex run per conversation/thread at a time
- File exchange:
  - Upload files from desktop/mobile into a conversation
  - Receive/download files generated by Codex
- VPS-friendly:
  - Runs via Docker Compose
  - Routed by Traefik with HTTPS
  - “Plug and play” install with minimal config
  - First-time user can deploy and use from mobile in minutes
- Safe-by-default architecture:
  - Codex app-server is **never exposed** to the public internet
  - Only the backend talks to Codex over localhost/bridge
- Modular and scalable foundation:
  - Clear separation of web/api/worker concerns
  - Stateless API containers (horizontal scale later)
  - Background jobs can move to dedicated worker process without redesign

### Nice-to-have (post-MVP)
- Conversation search (search by title/message text)
- Heartbeat automations (scheduled instruction files)
- Slash commands (client-side mappings)
- Multiple workspaces / repos
- OAuth / Passkeys / TOTP 2FA
- Per-conversation “sandbox” isolation using containers/worktrees
- Background jobs + queue for long runs
- Team support (multi-user)

## High-level Architecture

### Services
1) **codexchat_front** (frontend service; `web` role)
- ChatGPT-like UI
- Talks to backend via HTTPS (REST) and WebSocket (streaming)

2) **codexchat_back** (backend service; `api` role)
- Handles auth, session, rate limits, lockouts
- Stores conversation metadata + messages
- Spawns / manages Codex app-server processes
- Bridges: browser WebSocket ↔ Codex (stdio or localhost ws)

3) **codexchat_db** (Postgres service; `db` role)
- Users, conversations, messages, lockout state (or Redis for lockouts)

3b) **storage** (MVP: local volume; later: S3-compatible object storage)
- Stores uploaded/generated files
- DB stores metadata + ownership/authorization links

4) **redis** (optional but recommended)
- Fast counters for login attempts + progressive bans
- WebSocket fanout if needed later

5) **worker** (optional in MVP, first-class in MVP+)
- Runs scheduled/async jobs (heartbeat automations, long tasks)
- Same private access pattern to Codex as API

6) **traefik**
- TLS termination + routing to web/api
- Optional basic rate limits at edge for /login and websocket upgrade

### Data flow
Browser → Traefik → API
API ↔ Codex app-server (PRIVATE)
API → DB/Redis
API ↔ Storage (files)
Worker → DB/Redis → Codex app-server (PRIVATE)

**Important:** Codex app-server must bind to `127.0.0.1` or an internal Docker network only.

## Security Model (MVP)

### Authentication
- Email + password
- Password hashing: Argon2id (preferred) or bcrypt (strong params)
- Sessions:
  - httpOnly secure cookies
  - CSRF protection (if using cookies)

### Lockouts / brute force protection
Progressive lockout keyed by `(email, ip)` and also `(ip)`:

Example policy:
- 3 failures → 5 min ban
- next threshold → 30 min ban
- next threshold → 6 hours ban
- next threshold → 1 week ban

Notes:
- Reset failure count on successful login
- Record `last_failed_at`, `fail_count`, `ban_until`, `ban_level`
- Always return generic login errors (do not reveal if email exists)

### Transport + exposure
- Only expose:
  - `web` (UI)
  - `api` (auth + websocket)
- Do NOT expose Codex app-server to the internet.
- Traefik provides HTTPS with Let’s Encrypt.

## Codex Integration (MVP)

### Principle
The API owns the Codex process lifecycle.
MVP process model: **spawn-per-turn**.

### Mode
Prefer **stdio** for stability (recommended):
- API spawns: `codex app-server --listen stdio://`
- API communicates via JSON-RPC frames over stdin/stdout
- API publishes output events to the browser websocket

Alternative (optional): Codex ws mode
- `codex app-server --listen ws://127.0.0.1:PORT`
- Still only reachable from inside the VPS/network

### Threading model
- Each conversation maps to a Codex `thread_id`
- “New chat”:
  - create conversation in DB
  - start a new codex thread and store `thread_id`
- “Resume chat”:
  - load conversation + `thread_id`
  - spawn a fresh codex app-server process for that turn
  - continue with a new “turn” using stored `thread_id`
- Process lifecycle (MVP selected):
  - spawn Codex app-server per message turn
  - stream response
  - persist outputs
  - terminate process after turn completion
- Concurrency rule:
  - enforce a per-conversation lock while a Codex run is active
  - reject/queue concurrent sends to the same conversation until run completes

### Workspace model (MVP)
Simplest version:
- One configured workspace path (mounted into `api` container)
- All conversations operate on that workspace

Later:
- One workspace per conversation (git worktree or container copy)

### Conversation list
- Sort by `updated_at DESC`
- Title:

- Optional MVP+ search:
  - Search conversations by title and message text
  - Server-side query for large histories

## Heartbeat Automations (MVP+)

### Goal
Allow users to define one or more schedules where a specific markdown file is sent to Codex automatically, so Codex can act on that instruction repeatedly.

### Core behavior
- User creates a heartbeat job linked to:
  - `conversation_id`
  - `instruction_file_path` (example: `/workspace/ops/heartbeat.md`)
  - one or more intervals (example set: `5m`, `10m`, `15m`, `30m`, `60m`)
  - enabled/disabled flag
- On each trigger:
  - worker reads current markdown file contents
  - sends file contents as a user turn into the mapped conversation/thread
  - persists assistant output in normal message history
- Defaults:
  - Heartbeat jobs are disabled by default
  - Each user can create up to `10` heartbeat jobs by default
  - User can set heartbeat job cap to unlimited in settings

### Data model (minimal)
- `heartbeat_jobs`
  - id, user_id, conversation_id, instruction_file_path, enabled, created_at, updated_at
- `heartbeat_schedules`
  - id, heartbeat_job_id, interval_minutes, next_run_at, last_run_at
- `heartbeat_runs`
  - id, heartbeat_job_id, started_at, finished_at, status, error_text

### Safety and reliability
- Enforce minimum interval (example: 5 minutes)
- Prevent overlap for same job (skip or queue if prior run still active)
- Retry with capped backoff on transient failures
- Log each run for debugging and auditability

### Execution mode policy
- Support `regular` and `yolo` modes.
- YOLO mode is available to all users.
- Show explicit risk warnings before enabling/using YOLO mode.

## Traefik Routing (MVP)

### Network requirement
- Reuse an existing external Traefik Docker network.
- Do not create a new Traefik network in this project.
- In Compose, mark the Traefik network as `external: true` and attach `web`/`api` to it.

### Hostnames
- `APP_DOMAIN` → frontend
- `API_DOMAIN` (optional) → backend
Or:
- same domain, route by path:
  - `/` → web
  - `/api/*` and `/ws` → api

### TLS
- Let’s Encrypt (HTTP-01) via Traefik

### Edge rate limits (recommended)
- Add a Traefik middleware to limit:
  - `POST /auth/login`
  - websocket upgrades `/ws`

App-layer lockouts still required.

## Installation UX (Plug & Play)

### Target experience
1) Clone repo
2) Run `./scripts/setup.sh`
3) Script verifies prerequisites (Docker, Codex install/login) and can install Codex CLI if missing
4) Script asks for access mode:
   - `traefik` (recommended/production)
   - `ip` (dev/test only, with explicit warning)
5) Script bootstraps `.env`, auto-detects Traefik, and can install managed Traefik when missing
6) Visit domain (or dev/test IP mode), login, start chatting

### User-side setup story
1) User runs a single install sequence on VPS (`git clone`, `./scripts/setup.sh`)
2) Setup can install Codex CLI if missing and requires Codex login before proceeding
3) Setup configures access mode (Traefik or IP dev/test) and networking
4) App boots with `codexchat_front`/`codexchat_back`/`codexchat_db` (and optional redis/worker)
5) User opens app URL from phone or desktop browser
6) User logs in, creates a new chat, resumes old chats later
7) User uploads files into chat and downloads files returned/generated by Codex
8) Optional: user configures heartbeat job(s) for recurring markdown instructions

### Admin bootstrap
Pick one:
- Seed a single admin user via env on first run
- Or provide a CLI: `docker compose exec codexchat_back ./bin/create-user`

Recommended default:
- First user is created via SSH/VPS shell and is admin.
- Public self-registration remains disabled by default.
- Admin creates additional users from the web interface (Settings -> Admin).

## Acceptance Criteria (MVP)

### Auth + lockouts
- Correct password logs in
- Wrong password:
  - increments failure count
  - locks after policy thresholds
  - respects ban duration and escalation
- Can’t enumerate emails via error messages

### Conversations
- Create new conversation
- Conversation list shows it
- Messages persist in DB
- Resume conversation:
  - loads message history
  - continues context (assistant responds in-thread)
- Desktop + mobile both support:
  - start new conversation
  - resume old conversation
  - stable sidebar navigation pattern per form factor
- MVP+ search works for conversation title/message text
- Shared visibility is intentional (all users can access all conversations/files on same VPS)
- Concurrent sends to the same conversation are blocked or queued while active run lock exists

### File exchange
- User can upload attachments from desktop and mobile
- Uploaded files are stored under an app-managed files directory
- Uploaded files are available to Codex in the correct conversation context
- On send, backend passes file path(s) to Codex alongside the user message content
- User can download assistant-generated/shared files with auth checks
- Default upload max size is `15 MB` and user-adjustable in settings

### Chat UX
- Streaming assistant output works smoothly
- Code blocks render properly
- Sidebar navigation is instant and stable
- UI is fully responsive and usable on modern mobile devices

### Deployment
- Traefik routes both web and api with TLS
- No public ports for Codex app-server
- Container restart does not lose DB or conversations
- Container restart does not lose uploaded files (persistent volume)
- Optional worker service can be enabled for heartbeat jobs without changing core architecture
- Production path uses custom domain + HTTPS; IP-only mode is explicitly dev/test only
